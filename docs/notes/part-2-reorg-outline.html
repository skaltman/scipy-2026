Part 2 reorg — outline & plan

Working doc for restructuring the second half of the talk (the “LLMs are still useful for data analysis” section). Goal: open it up from “here’s Posit Assistant” to “here’s how to design any data-analysis agent,” with Posit Assistant as the worked example.

Audience (SciPy)

SciPy 2026 = 25th annual Scientific Computing with Python conference (Minneapolis, July 2026). Attendees are researchers, developers, and industry folks building open-source Python tools for science/engineering/data. Relevant tracks include Data-Driven Discovery / ML / AI.

Implications for framing:

  • This audience builds their own tools — many will be building or wiring up their own agents, not just using an off-the-shelf one. Lean into that.
  • This audience already has domain code they trust (their own packages, well-tested numpy/scipy/pandas pipelines). That’s exactly the setup where the “trusted code” point lands hardest: giving the agent your vetted code beats letting it reinvent it.

The two lessons — prevention vs. mitigation

The old framing was “1. Tell them to care about being right / 2. Design the scaffold so it matters less if they’re wrong.” Problem: “tell them” only covers prompting, but lots of the harness (including code as a foundation) also lowers the mistake rate — so lessons 1 and 2 blurred.

New framing:

  1. Help them get it right — prevention. Everything in the harness that lowers the chance of a mistake: prompting them to value correctness, giving them the right tools / session access, and giving them trusted code to reuse.
  2. Make it matter less when they’re wrong — mitigation. Auditable + reproducible code they write, a shared environment, and (our choice) pausing often.

This split also dissolves the “code as the foundation” blur by giving code two distinct, clearly-placed roles:

  • Trusted code you hand them → prevention → lesson 1
  • Auditable / reproducible code they write → mitigation → lesson 2

Core reframes

  1. Open it up to agents generally. Add a “whether you use an agent like Posit Assistant or build your own” beat. Can appear in multiple places (harness slide, and echoed on the final summary card).

  2. Separate general principles from Posit-specific choices. “Code as a foundation” (near-universal) and “pause often” (our specific call) currently sit at the same level. Re-parent so the principle is the headline and the choice is the example.

  3. Trusted code = the prevention facet of “Code as the Foundation.” If you have code you trust to do a data-science task (complex or not), have the agent call it instead of writing its own. Often better than model- authored code. Lands especially well with a SciPy audience that already maintains trusted scientific code.

  4. Shared environment is a coding-agent principle, not an all-agent one. Applies to many/all coding agents, but not necessarily every agent. Keep it, but don’t over-generalize.

  5. “Pause often” is our choice, not a rule. Re-parent under a general principle: design for how people will actually use the agent, and what that means for correctness/reproducibility. For data analysis the real goal is understanding, so we made Posit Assistant pause often.

Internal note — do NOT present

The trusted-code idea draws partly from Commons (an internal coding-agent package, ~/GitHub/rstudio/commons). Commons is not public, so it must not be named or described in the talk. The mechanism there: a semantic layer — a pool of pre-vetted, trusted queries/definitions that are the agent’s “happy path,” so it reuses trusted artifacts instead of authoring its own SQL, only writing its own when nothing covers the request. Present this generically as “trusted code.”
